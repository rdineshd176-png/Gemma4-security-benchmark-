$$C_i = P_i \oplus K_i$$

$$C_i \oplus C_{i+L} = (P_i \oplus K_i) \oplus (P_{i+L} \oplus K_i) = P_i \oplus P_{i+L}$$

*   **Similarity** $= 1 - \text{Normalized Distance}$

*   **If $L$ is incorrect:** The distance is often higher (e.g., 0.6 or 0.7), indicating poor correlation.

Therefore, the correct key length is the one that yields the **minimum normalized distance**.

#### 1. The Pickle Vulnerability (The Threat)
Pickle is a serialization format that is inherently unsafe because it is not just a data format; it is a **code execution format**. When `pickle.loads()` runs, it doesn't just reconstruct data; it executes instructions (opcodes) embedded in the byte stream.

#### 2. The HMAC Defense (The Solution)
The HMAC-SHA256 signature provides **integrity** and **authenticity**.

*   **Integrity:** It guarantees that the data has not been altered since it was signed.
*   **Authenticity:** It guarantees that the data was signed by someone who possesses the secret key.

When the server receives the cookie, it performs these steps:
1. Takes the received data (the pickled bytes).
2. Uses the secret key to calculate a new HMAC-SHA256 hash of the data.
3. Compares this calculated hash to the signature provided in the cookie.

If the attacker modifies the pickled data (e.g., injecting their malicious code), the server's calculated hash will **not** match the signature, and the server will reject the data *before* calling `pickle.loads()`.

An attacker can exploit this vulnerability by running a concurrent process that modifies the file pointed to by `path` immediately after the `access()` call returns success, but before the `open()` call executes.

2. **Attacker Action (Race):** The attacker's concurrent process monitors the file and, immediately after the `open_file_safe` function executes `access(path, R_OK)` (which returns success because `safe_config.txt` is readable), the attacker quickly deletes `safe_config.txt` and replaces it with a symbolic link pointing to a sensitive target (e.g., `/etc/shadow` or `/var/www/html/admin_secrets.php`).

4. **Result:** The `open()` system call follows the newly created symbolic link, and the function successfully returns a file descriptor (`fd`) pointing to the sensitive file (`/etc/shadow`), even though the initial check was performed on the benign file. ### 2. Permission/State Change Attack

The attacker can change the file's permissions or content state.

1. **Initial State:** The file is readable (`rw-r--r--`).
2. **Attacker Action (Race):** The attacker changes the file permissions to be unreadable (e.g., `chmod 000 filename`).
3. **Function Execution (Use):** The `open_file_safe` function proceeds to `open(path, O_RDONLY)`.
4. **Result:** The `open()` call fails (or returns an FD that is immediately unusable), even though the initial `access()` check passed. This can be used to cause a Denial of Service (DoS) or to trigger unexpected logic in the calling code.

---

## ✅ Mitigation Strategy

The most robust way to mitigate TOCTOU is to eliminate the gap between the check and the use, or to use system calls that perform both actions atomically.

**Recommended Fix (Using `open()` and `fstat()`):**

Instead of relying on `access()`, the function should attempt the `open()` first. If `open()` succeeds, it gets a file descriptor (`fd`). Then, it should use `fstat()` on that descriptor to check the permissions of the *actual* opened file, guaranteeing that the check and the use refer to the same file state.

```c
#include <sys/stat.h>
#include <unistd.h>
#include <fcntl.h>

int open_file_safe_mitigated(char *path) {
    // 1. Attempt the use first (open)
    int fd = open(path, O_RDONLY);
    
    // Check if the open itself failed (e.g., file doesn't exist)
    if (fd < 0) {
        return -1;
    }

    // 2. Check the permissions of the file descriptor (fstat)
    struct stat st;
    if (fstat(fd, &st) == -1) {
        // Failed to stat the file, close the FD and fail
        close(fd);
        return -1;
    }

    // Check if the file is readable (S_IRUSR | S_IRGRP | S_IROTH)
    // This checks if *any* user/group/other has read permission.
    if (!(st.st_mode & (S_IRUSR | S_IRGRP | S_IROTH))) {
        // File exists and opened, but permissions are insufficient
        close(fd);
        return -1;
    }

    // Success: The file is open AND the permissions were verified atomically
    return fd;
}
```

## 🔍 Vulnerability Identification

**Vulnerability:** Double Free
**Location:** The calls to `free(buf)` in `free1()` and `free2()`.

The pointer `buf` is allocated memory once in `alloc()`. It is then passed to `free1()`, which releases the memory block back to the heap manager. When `free2()` is called, it attempts to release the *exact same* memory block again.

## 🧠 Explanation of the Vulnerability

When a memory block is freed, the heap manager (like `ptmalloc` in glibc) does not simply erase the memory; it updates internal metadata about that block. This metadata includes:

1.  The size of the chunk.
2.  Pointers indicating where the chunk sits in the heap's free lists (e.g., the `fd` and `bk` pointers).

When `free1()` runs, the heap manager marks the block as free and inserts it into a free list (often the TCache or a fast bin).

When `free2()` runs, the heap manager sees that the block is *already* marked as free. It attempts to process the block again, leading to heap corruption. Depending on the specific heap implementation, this corruption can manifest in several ways:

*   **Immediate Crash:** The heap manager detects the double free immediately and aborts the program (the safest outcome).
*   **Metadata Corruption:** The heap manager attempts to update the pointers of the chunk again, potentially overwriting the `fd` pointer of the *previous* chunk in the free list, or overwriting the chunk's own header data.
*   **Silent Corruption:** The program continues running, but the heap state is now inconsistent, making future memory operations unpredictable.

## 🚀 Exploitation Path

An attacker exploits a double free by leveraging the corrupted heap state to achieve **Arbitrary Write** or **Control Flow Hijacking**.

### Step 1: Heap Corruption (The Double Free)

1.  **`alloc()`:** Allocate chunk $B_1$.
2.  **`free1()`:** Free $B_1$. The heap manager adds $B_1$ to the free list, setting its internal pointers (e.g., $B_1 \rightarrow fd$ points to the next free chunk, $B_1 \rightarrow bk$ points to the previous free chunk).
3.  **`free2()`:** Free $B_1$ again. The heap manager attempts to insert $B_1$ into the free list a second time. This insertion process often involves updating the pointers of the chunk *before* $B_1$ in the list, causing a pointer overwrite.

### Step 2: Re-Allocation and Pointer Duplication

The attacker now performs subsequent allocations:

1.  **`malloc(size)` (Allocation A):** The heap manager returns $B_1$. The attacker now has a pointer $P_A$ pointing to $B_1$.
2.  **`malloc(size)` (Allocation B):** Because $B_1$ was listed twice, the heap manager returns $B_1$ *again*. The attacker now has a second pointer $P_B$ pointing to the exact same memory location $B_1$.

### Step 3: Arbitrary Write and Control Flow Hijacking

The attacker can now use $P_A$ and $P_B$ to perform a "write-what-where" attack:

1.  **Write Data:** The attacker writes a specific value (e.g., the address of a function like `system()`) using $P_A$.
2.  **Overwrite Critical Data:** This data is written into $B_1$. Since $P_B$ points to the same location, the attacker knows exactly where the data is.
3.  **Trigger Execution:** The attacker then forces the program to use the memory block $B_1$ in a way that triggers a function call (e.g., casting $B_1$ to a function pointer and calling it, or using $B_1$ as a Virtual Table pointer).

**Result:** The program executes the function at the address the attacker wrote (e.g., `system(address_of_shellcode)`), achieving **Remote Code Execution (RCE)**.

An attacker can exploit this vulnerability by manipulating the `id` parameter in the URL.

**Goal:** Retrieve all users, regardless of their ID.
**Payload:** `id=1 OR 1=1`
**URL:** `http://example.com/page.php?id=1 OR 1=1`

**Resulting Query:**
```sql

```


**Payload:** `id=1; DROP TABLE users; --`


1. `SELECT name, email FROM users WHERE id=1`
2. `DROP TABLE users`

The primary defense against SQL Injection is to ensure that user input is always treated as **data**, never as executable code.

### 1. Use Prepared Statements (The Best Practice)

```php
$stmt->execute();

// 4. Get the results
$result = $stmt->get_result();
```

```php
$stmt = $pdo->prepare("SELECT name, email FROM users WHERE id=?");

$result = $stmt->fetchAll();
```

```php
$safe_id = mysqli_real_escape_string($mysqli, $_GET['id']);
```

## 🛡️ Vulnerability Identified

**Vulnerability:** Use of Electronic Codebook (ECB) Mode.
**Type:** Lack of Semantic Security / Deterministic Encryption.

### Explanation of the Vulnerability

ECB mode is the simplest mode of operation for AES, but it is highly insecure for encrypting data where the same block of plaintext may appear more than once.

1. **Deterministic Output:** In ECB mode, the encryption of a plaintext block is entirely independent of all other blocks. The cipher uses the same key and the same algorithm for every block.
2. **The Flaw:** If the same 16-byte block of plaintext appears multiple times in the input data, it will **always** produce the exact same 16-byte block of ciphertext.
3. **Information Leakage:** This deterministic nature leaks significant structural information about the data. An attacker does not need to decrypt the entire message to understand its patterns, repetitions, and structure.

---

## 🔪 Exploitation Path

An attacker observing the ciphertext can exploit the deterministic nature of ECB to perform several attacks without ever knowing the secret key.

### 1. Pattern Recognition and Data Structure Analysis

The attacker can scan the ciphertext and immediately identify repeated blocks.

*   **Example:** If the plaintext is a list of records:
    *   `[Header Block] [User A Block] [User B Block] [User A Block] [User C Block]`
    *   The resulting ciphertext will be:
        *   `[C_Header] [C_A] [C_B] [C_A] [C_C]`
*   **Exploitation:** The attacker instantly knows that "User A" appears twice, and they can deduce the relative positions of the records, even if they don't know what "User A" means.

### 2. Block Swapping (Reordering)

Since each block is independent, an attacker can swap two ciphertext blocks ($C_A$ and $C_B$) and the corresponding plaintext blocks ($P_A$ and $P_B$) will be swapped without needing to decrypt the message.

*   **Scenario:** An attacker intercepts a message containing two blocks:
    *   $P_A$: "Amount: \$100"
    *   $P_B$: "Type: Debit"
    *   Ciphertext: $[C_A, C_B]$
*   **Attack:** The attacker swaps them to create $[C_B, C_A]$.
*   **Result:** The decrypted message becomes: "Amount: \$100" (but the order is now $P_B$ followed by $P_A$), which might be interpreted as "Type: Debit, Amount: \$100" (depending on the application logic). This is a classic way to manipulate data integrity.

### 3. Known Plaintext Attack (KPA)

If the attacker knows a specific plaintext block ($P_X$) that is likely to appear in the message (e.g., they know the message must contain the block "NULL" or "True"), they can search the ciphertext for the exact corresponding block ($C_X$).

*   **Exploitation:** If the attacker knows $P_X = \text{"True"}$, they search the ciphertext for $C_X$. If they find it, they have confirmed that the plaintext "True" exists at that specific location, even if the rest of the message is gibberish.

---

## ✅ Recommended Fix

The primary fix is to switch the mode of operation from ECB to a secure, chained mode that incorporates an Initialization Vector (IV).

**Recommended Mode:** **AES.MODE_CBC** (Cipher Block Chaining) or **AES.MODE_GCM** (Galois/Counter Mode).

**Why GCM is preferred:** GCM is an Authenticated Encryption with Associated Data (AEAD) mode. It not only encrypts the data but also generates a **Message Authentication Code (MAC)** or **Authentication Tag**. This means that if an attacker swaps, reorders, or modifies even a single bit of the ciphertext, the decryption process will fail the authentication check, immediately alerting the recipient that the data has been tampered with.

### Code Fix Example (Using CBC)

```python
from Crypto.Cipher import AES
import os # Used to generate a random IV

def encrypt_data_secure(data, key):
    # 1. Generate a random Initialization Vector (IV)
    iv = os.urandom(16) 
    
    # 2. Initialize the cipher in CBC mode
    cipher = AES.new(key, AES.MODE_CBC, iv)
    
    # 3. Encrypt the padded data
    ciphertext = cipher.encrypt(pad(data, 16))
    
    # 4. Return the IV along with the ciphertext (IV must be transmitted with the data)
    return iv + ciphertext 
```

## 🛡️ Vulnerability Identified: Weak Randomness / Predictability

The function uses Python's built-in `random` module. While the `random` module is excellent for general-purpose simulations and statistical sampling, it is a **Pseudo-Random Number Generator (PRNG)**, typically based on the Mersenne Twister algorithm.

A PRNG is deterministic. This means that the sequence of "random" numbers is not truly random; it is generated by a mathematical formula starting from an initial value called a **seed**. If an attacker can determine the internal state (the seed) of the PRNG, they can predict every single token that will be generated subsequently.

### Why this is a vulnerability:

1. **Deterministic Output:** The tokens are not truly unpredictable.
2. **State Recovery:** The Mersenne Twister algorithm has a relatively small internal state (624 32-bit integers). It is computationally feasible for an attacker to observe a small number of outputs (often as few as 50-100 tokens) and use algorithms (like those used in cryptanalysis) to reconstruct the exact internal state of the generator.

---

## 🔪 Exploitation Path (The Attack)

The attacker does not need to brute-force the entire token space ($62^{32} \approx 1.86 \times 10^{57}$ possibilities). Instead, they exploit the deterministic nature of the generator.

### Step 1: Observation (Data Collection)
The attacker observes a set of tokens generated by the system. For example, they might observe the tokens generated when a user registers, or when a password reset link is created.

*   **Observed Data:** $T_1, T_2, T_3, \dots, T_{50}$

### Step 2: State Reconstruction (The Attack Core)
The attacker feeds this observed sequence of tokens into a specialized algorithm (e.g., a function that implements the MT state recovery algorithm). This algorithm works backward from the observed outputs to determine the exact values of the 624 internal variables that define the current state of the PRNG.

*   **Result:** The attacker knows the exact internal seed/state of the generator at the moment $T_1$ was created.

### Step 3: Prediction (The Payoff)
Once the internal state is known, the attacker simply runs the same PRNG algorithm locally. They can then calculate the next token ($T_{51}$), the token after that ($T_{52}$), and so on, with near-perfect certainty.

*   **Scenario 1 (Targeted Guessing):** If the system is about to generate a token for a high-value account, the attacker can predict it instantly.

### Why MD5 is Vulnerable

1. **Speed (Lack of Computational Cost):** MD5 is designed to be extremely fast. While speed is good for data integrity checks, it is terrible for password storage because it allows an attacker to compute billions of hashes per second using modern hardware (especially GPUs).
2. **Collision Susceptibility:** MD5 is known to be vulnerable to collision attacks (where two different inputs produce the same hash). While this is less critical for password storage than speed, it means an attacker could potentially find two different passwords that map to the same stored hash.
3. **Lack of Salting (The Critical Flaw):** The code hashes the password directly (`hashlib.md5(password.encode())`). It does not incorporate a unique, random **salt**. Without a salt, if two users choose the same password (e.g., "password123"), they will have the exact same hash. This allows attackers to use pre-computed tables to their advantage.

---

## 🔪 Exploitation Path

An attacker can exploit the weakness of MD5 and the lack of salting through two primary methods:

### 1. Rainbow Table Attack (Exploiting Lack of Salt)

* **Process:** An attacker pre-computes a massive database of common passwords and their corresponding MD5 hashes (a "Rainbow Table").
* **Exploitation:** The attacker retrieves the list of hashes from the `users` table. For each stored hash, they simply look it up in their pre-computed Rainbow Table. If the hash matches, they instantly know the original password, without needing to run any brute-force calculations.
* **Impact:** If the database contains many users with common passwords, the attacker can crack hundreds or thousands of accounts almost instantly.

### 2. Brute-Force Attack (Exploiting Speed)

* **Process:** If the attacker does not have a Rainbow Table, they can launch a brute-force attack. They take a list of potential passwords (e.g., dictionary words, common combinations) and run them through the MD5 function.
* **Exploitation:** Because MD5 is so fast, an attacker using a GPU can test millions of passwords per second. They compare the hash of their guessed password against the stored hash in the database. When they find a match, they have successfully cracked the password.
* **Impact:** Even if the password is not common, the attacker can rapidly guess and verify it, leading to account compromise.

This code snippet utilizes the Data Encryption Standard (DES) cipher in Electronic Codebook (ECB) mode. While DES itself is considered cryptographically weak today (due to its 56-bit key size), the choice of **ECB mode** is the primary and most significant vulnerability in this specific implementation.

---

## 🛡️ Vulnerability Identification

**Vulnerability:** Deterministic Encryption / Lack of Semantic Security (specifically due to using ECB mode).

**Description:** ECB mode encrypts each block of plaintext independently. If the same block of plaintext appears multiple times in the input data, it will *always* produce the exact same block of ciphertext. This deterministic nature leaks information about the plaintext structure, allowing an attacker to deduce patterns, repetitions, and relationships without ever knowing the key.

---

## 🔍 Explanation of the Vulnerability

In a secure cipher mode (like CBC, CTR, or GCM), the encryption of a block depends not only on the block itself but also on the previous block (chaining). This ensures that even if the same plaintext block appears twice, the chaining mechanism will modify it slightly, resulting in two different ciphertext blocks.

In ECB mode, the process is simple:

$$
\text{Ciphertext Block}_i = \text{Encrypt}(\text{Plaintext Block}_i, \text{Key})
$$

**Example:**
If the plaintext is: `[Header] [User ID 123] [Header] [User ID 456]`

The ciphertext will be: `[C_Header] [C_123] [C_Header] [C_456]`

The attacker immediately sees that `C_Header` is repeated, confirming that the word "Header" appeared twice.

---

## 🚀 Exploitation Path

An attacker who intercepts the ciphertext (`ct`) can exploit the deterministic nature of ECB in several ways:

### 1. Pattern Recognition (Deduction)
The attacker can analyze the ciphertext to identify repeated blocks. This is highly effective when encrypting structured data (like database records, financial transactions, or image data).

*   **Exploitation:** If the attacker knows that a specific block of ciphertext (`C_A`) corresponds to a known plaintext block (e.g., "Transfer Amount: $100"), they can scan the entire ciphertext to find every instance of `C_A` and immediately know that "Transfer Amount: $100" occurred at that position.

### 2. Reordering/Swapping (Manipulation)
Since each block is independent, the attacker can rearrange the blocks of the ciphertext without needing to decrypt them.

*   **Exploitation:** Imagine a financial transaction record encrypted as:
    $$
    \text{CT} = [\text{Block}_{\text{Sender}}] [\text{Block}_{\text{Amount: 100}}] [\text{Block}_{\text{Recipient}}]
    $$
    The attacker could swap the middle two blocks to create a new ciphertext:
    $$
    \text{CT}_{\text{New}} = [\text{Block}_{\text{Sender}}] [\text{Block}_{\text{Recipient}}] [\text{Block}_{\text{Amount: 100}}]
    $$
    When the recipient decrypts $\text{CT}_{\text{New}}$, the plaintext will be:
    $$\text{PT}_{\text{New}} = [\text{Sender}] [\text{Recipient}] [\text{Amount: 100}]$$
    The meaning of the data has been changed simply by reordering the blocks.

### 3. Block Injection (Tampering)
The attacker can inject a known, valid ciphertext block into a stream of data, effectively inserting a known piece of information into the plaintext stream.

*   **Exploitation:** If the attacker wants to force a transaction to be marked as "Urgent," they simply take the ciphertext block corresponding to "Urgent" ($\text{C}_{\text{Urgent}}$) and insert it anywhere in the data stream. The recipient will decrypt it and see "Urgent" appear at that location.

---

## ✅ Mitigation (How to Fix It)

The primary fix is to change the mode of operation.

**Recommendation:** Use **Cipher Block Chaining (CBC)** or, even better, **Galois/Counter Mode (GCM)**.

**Example using CBC:**

```python
from Crypto.Cipher import DES
# 1. Generate a random Initialization Vector (IV)
iv = os.urandom(8) 
# 2. Initialize the cipher using the IV and CBC mode
cipher = DES.new(key, DES.MODE_CBC, iv) 
ct = cipher.encrypt(data)
# The IV must be transmitted along with the ciphertext (ct)
```

By using CBC, the encryption of the first block depends on the IV, and the encryption of every subsequent block depends on the previous ciphertext block, ensuring that even identical plaintext blocks produce unique ciphertext blocks.

## 🔍 Vulnerability Analysis

### 1. Specific Vulnerability: Server-Side Template Injection (SSTI)

SSTI occurs when an application takes user-supplied input and embeds it directly into a template string without proper sanitization or separation. The template engine (in this case, Jinja2, which Flask uses by default) then interprets the injected input as executable template code, allowing the attacker to run arbitrary Python code on the server.

### 2. Why the Code is Vulnerable

The vulnerability lies in this line:
```python
return render_template_string('Hello ' + name)
```

By using string concatenation (`+ name`), the attacker's input (`name`) is treated as part of the template structure itself, rather than just a simple string value. Jinja2 will parse the entire resulting string. If the attacker injects Jinja2 syntax (like `{{ ... }}`), the server will execute the code inside the braces.

## 🚀 Exploitation Path

An attacker can exploit this vulnerability by providing a malicious payload in the `name` query parameter.

### Step 1: Crafting the Payload

A simple payload to prove SSTI is to make the server calculate a mathematical expression.

**Payload:** `{{ 7 * 7 }}`

### Step 2: Executing the Attack

The attacker accesses the endpoint with the payload:

**URL:** `http://[your-server]/hello?name={{ 7 * 7 }}`

### Step 3: Server Execution and Result

1.  The Python code executes: `name` is set to `{{ 7 * 7 }}`.
2.  The template string becomes: `'Hello {{ 7 * 7 }}'`.
3.  `render_template_string` processes this string.
4.  Jinja2 executes the code `7 * 7`.
5.  **The server returns the HTML:** `Hello 49`

### Advanced Exploitation (Code Execution)

An attacker can escalate this to full Remote Code Execution (RCE) by accessing internal Python objects available in the Jinja2 environment.

**Payload Example (Accessing Flask Configuration):**
```
{{ config.items() }}
```
**Result:** The server will return a list of all key-value pairs currently configured in the Flask application (e.g., `{'SECRET_KEY': '...'}`).

**Payload Example (Executing OS Commands):**
By accessing the `__import__` function and then the `os` module, the attacker can run system commands.

**Payload:**
```
{{ __import__('os').system('ls -la') }}
```
**Result:** The server executes the `ls -la` command on the host machine and returns the output of that command directly in the HTML response.

---

## ✅ Mitigation

The best way to mitigate SSTI is to **pass the user input as a variable to the template function**, allowing Jinja2 to handle the injection and ensure the input is treated as data, not code.

**Vulnerable Code:**
```python
return render_template_string('Hello ' + name)
```

**Secure Code (Recommended Fix):**
```python
# Pass 'name' as a variable to the template string
return render_template_string('Hello {{ name }}', name=name)
```

By using `{{ name }}`, the input is automatically escaped and interpreted as a variable placeholder. If an attacker inputs `{{ 7 * 7 }}`, the template engine will render the literal string `{{ 7 * 7 }}` (or just `7 * 7`), rather than executing the code, thus preventing the injection.
